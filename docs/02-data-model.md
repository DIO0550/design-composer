# 02. データモデル

## 基本原則

- 部品も画面も「プリミティブのツリー」という同一モデルで表現する。
  - **画面** = ルートのアートボードに並べたツリー
  - **部品** = ツリー内で命名して再利用可能にしたサブツリー
- **配置は座標ではなく、ツリー構造とレイアウトプロパティで決まる**（実HTML/CSSのフローレイアウト）。要素の編集は「どこに置くか」ではなく「どの構造に入れるか」。GUIのドラッグ操作もツリー内の移動として解釈する。
  - overlay 系（モーダル・バッジ等）は将来、親相対の配置を縛ったプリミティブで対応する。キャンバス座標は導入しない。

## ノード

```json
{
  "name": "login-form",
  "type": "Box",
  "props": { "direction": "column", "gap": "md" },
  "children": [ ... ]
}
```

| フィールド | 型 | 必須 | 内容 |
|---|---|---|---|
| `name` | string | ✔ | ドキュメント全体で一意な識別子（01-file-format「ノードの識別」参照） |
| `type` | string | ✔ | プリミティブ名（語彙は 03-schema で定義） |
| `props` | object | - | プロパティ。省略時はすべてデフォルト |
| `children` | array | - | 子ノード。子を持てるかは type のスキーマが決める |

## props

### 値の形: フラットなスカラーのみ

- props の値は **string / number / boolean のみ**。オブジェクト・配列は許さない
- 複合的な値（影・ボーダー等）は props に構造として持たず、**トークン側に実体を1回だけ定義し、props はトークン名で参照する**
  - 例: `"shadow": "sm"` — x/y/blur 等の実体は tokens.shadows.sm にある
- CSSショートハンド相当は個別プロパティに平坦化する（`borderWidth` / `borderColor` 等）
- この制約により、スキーマ定義は「1 prop = 型・ドメイン・デフォルト」の一様な辞書、プロパティパネルは「1 prop = 1 コントロール」、AI編集は1階層の代入で完結する

### 値のドメイン: 3種類

| ドメイン | 例 | 語彙を決めるのは |
|---|---|---|
| enum | `direction: "row" \| "column"` | スキーマ（固定） |
| トークン参照 | `gap: "md"`, `background: "primary"`, `shadow: "sm"` | ユーザー（tokens 定義から選ぶ） |
| 生リテラル | `width: 375`, `label: "ログイン"` | 自由入力（number / string） |

- **見た目に関わる値（色・余白・角丸・フォント・影など）は必ずトークン参照とし、生リテラルを許さない**（トークン縛り切り）
- 生リテラルは構造値（幅・高さ等の数値）と内容（テキスト等）に限る
- 各 prop がどのドメインに属するかはスキーマが定義する。1 prop は 1 ドメインのみ（混在なし）

### トークン参照の構文: 裸の名前

- トークン参照は `"gap": "md"` のように**トークン名をそのまま書く**。プレフィックス（`$` 等）やオブジェクト形式は使わない
- どのトークン種別を参照するかはスキーマ側が知っている（gap は spacing、background は colors）ため、値だけで曖昧にならない
- 存在しないトークン名への参照はスキーマ検証でエラーとする

### デフォルト値

- ドキュメントには**明示的に設定した props のみ**を保存する
- 未指定の props はスキーマ側で定義されたデフォルト値に従う
- 「書いてある = 意図がある」状態を保ち、diff から変更意図が読めるようにする

## 部品参照ノード

```json
{ "name": "save-button", "ref": "primary-button", "overrides": { "label": "保存" } }
```

- 部品と画面は別機能ではなく、同じ木構造に「命名して参照できる」層を1枚足したもの
- 参照はスキーマ上の明示的な参照ノードとして表現する。フォーマット側の参照機能には依存しない
- ファイル内参照のみ（詳細は 01-file-format）

### 部品のインターフェース（overrides の範囲）

部品側が「外から変更してよいもの」を **`publicProps` 宣言**として名前付きで公開する（Figma の Component Properties 相当）。

部品のルートはノードでもあるため、ノード自身の `props` と衝突しないよう、宣言は `publicProps` という別フィールドに持つ。

```json
"components": {
  "primary-button": {
    "publicProps": {
      "label": { "node": "primary-button-label", "prop": "content" }
    },
    "type": "Box",
    "props": { "background": "primary", "radius": "md" },
    "children": [
      { "name": "primary-button-label", "type": "Text", "props": { "content": "Button" } }
    ]
  }
}
```

- **インスタンスは宣言された publicProps しか上書きできない**。未宣言キーの overrides はバリデーションエラー
- 宣言は binding `{ node, prop }` で内部ノードの prop に結び付く
  - `node` は部品内部のノードの `name` を指す（全ノードが name を持つため、任意の内部ノードに binding できる）
- 上書き値のドメイン（enum / トークン / 生リテラル）は binding 先 prop のスキーマ定義をそのまま継承する。部品用の型システムは新設しない
- インスタンスのプロパティパネルは publicProps 宣言から自動生成する
- 部品の内部構造を変更しても、publicProps 宣言の binding を維持する限りインスタンスは壊れない
- ネストした部品（部品内の ref）の内部への binding は許さない。公開したいものは各部品が自分の publicProps として宣言し、親部品はそれに binding する（インターフェースの連鎖）
