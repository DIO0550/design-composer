# テスト規約

**基本方針: 古典学派(Classicist / Detroit School)でテストを書く。** 詳細は「古典学派のテスト」セクションを参照。

## 配置と命名

モジュールフォルダは `index.ts` を公開APIとし、テストは同フォルダ内の `__tests__/` に置く。

**基本形: 1フォルダ = `index.ts`(+ 必要なら実装ファイル1つ)+ `__tests__/`。** この形を必ず守ること(実装ファイルを分けるかどうかは `rules/architecture.md`「モジュールの公開API」参照。実装が1ファイルで完結する場合は `index.ts` に直接書く)。

```
<モジュールフォルダ>/
  index.ts                          # 公開API(外部はここからのみ import する)。実装が1ファイルで完結する場合はここに実装も書く
  __tests__/
    <対象ドメイン名>.<カテゴリ>.test.ts
```

- 実装ファイルを分けた場合、テストファイル名はその実装ファイル名に合わせる
- 実装ファイルが2つ以上必要になったらサブフォルダに分割し、各サブフォルダでも同じ形を保つ

命名規則: `{テスト対象ファイル}.{カテゴリ}.test.ts`

- カテゴリはテストの観点を表すラベル。例:
  - `layout.normal.test.ts` — 正常系
  - `layout.edge.test.ts` — 境界値・異常系
  - `layout.export.test.ts` — 特定の仕様領域
- **describe のネストの代わりにカテゴリ(=ファイル)で分割する**。1ファイルが肥大化したらカテゴリを分けること

## 古典学派のテスト

古典学派の2本柱は「**何をテストするか = 振る舞い**」と「**依存をどう扱うか = 実物を使う**」。両方を必ず守ること。

### 1. 振る舞い(仕様)をテストする

テストは**外から観測できる振る舞い**に対して書く。実装の内部構造に対して書かない。

- テスト対象は公開API(`index.ts` の export)の「入力 → 出力/最終状態」のみ。内部のヘルパー関数・中間データ構造・処理順序はテストしない
- テスト名は**仕様の文として**書く(「○○のとき□□になる」)。関数名・実装語彙ではなくドメイン語彙を使う
  - OK: `"要素を重ねて配置すると後から追加した要素が前面に表示される"`
  - NG: `"calculateZIndex が 2 を返す"`、`"parseLayout が内部で sortElements を呼ぶ"`
- 1テスト1振る舞い。複数の仕様を1つのテストで検証しない
- **リファクタリング耐性を判断基準にする**: 振る舞いを変えずに実装を書き換えたときに壊れるテストは、実装詳細をテストしている証拠なので書き直す
- テストケースの根拠は仕様・設計ドキュメントに求める。仕様にない振る舞いをテストで固定しない(実装の偶然の挙動を仕様化しない)

### 2. モックは原則使わない

依存は実物を使い、相互作用(呼び出し回数・引数)ではなく**結果の状態・戻り値を検証**する。

- `domains/` `services/` のテストでモック・スタブ・スパイの使用は**全面禁止**(純粋関数なので実物で検証できるはず。モックが必要に感じたら設計を見直す)
- features / hooks / components のテストでも、`domains/` `services/` は必ず実物を使う(ドメインロジックのモック化は禁止)
- 例外として代替してよいのは**プロセス外・制御不能な境界のみ**: `libs/` が包む外部I/O(Tauri API、localStorage、fetch、時刻、乱数)。この場合もモックライブラリによる自動モックより、テスト用の単純な実装(フェイク)を優先する
- `vi.mock` によるモジュール丸ごと差し替えは禁止
- 内部実装の呼び出し検証(`toHaveBeenCalledWith` 等)は外部境界の検証以外で使用しない

```typescript
// NG: ドメインロジックをモックして相互作用を検証
vi.mock("@/services/layout");
test("...", () => {
  render(<CanvasView />);
  expect(LayoutService.arrange).toHaveBeenCalledWith(/* ... */);
});

// OK: 実物を通して最終的な出力を検証
test("2つの要素を整列すると左端が揃って表示される", () => {
  render(<CanvasView layout={setupLayout({ /* ... */ })} />);
  expect(screen.getByTestId("element-1")).toHaveStyle({ left: "0px" });
});
```

## テストの書き方: ネスト禁止

Kent C. Dodds の "Avoid Nesting when you're Testing" に従う。

- `describe` のネストは禁止(ファイル直下に `test` / `it` をフラットに並べる。グルーピングが必要ならカテゴリファイルを分ける)
- `beforeEach` での変数共有・ミューテーションは禁止。テストに必要なセットアップは**各テスト内に書く**か、セットアップ関数(例: `setup()`, ファクトリ)を呼び出す
- テストは1つだけ読めば前提がすべて分かる状態にする(他のテストや上方のフックを読まないと理解できないテストは書かない)

```typescript
// NG
describe("Layout", () => {
  let layout: Layout;
  beforeEach(() => {
    layout = createLayout(/* ... */);
  });
  describe("when aligned", () => {
    it("...", () => { /* layout がどこで何になっているか追えない */ });
  });
});

// OK: __tests__/layout.normal.test.ts
test("要素を追加すると要素数が1増える", () => {
  const layout = setupLayout({ /* このテストに必要な状態を明示 */ });
  expect(Layout.count(Layout.addElement(layout, element))).toBe(1);
});
```

- クリーンアップが必要な場合のみ `afterEach` を許可(リソース解放など、テストの理解に影響しないもの)
